AI System Design for PMs Shifts Loop Context What PMs Own Failures Learn
Essay · Research-backed · 2026

AI system design isn't just system design with LLMs bolted on.

If you believe it is, you're already building it wrong. A field guide to the four shifts, the agentic loop, and the three things AI product managers must own — written for PMs who came up through traditional product work and feel a little behind.

10 min read Synthesized from 12+ primary sources Anthropic · LangChain · ArXiv · OpenTelemetry
Scroll

I spent the last few weeks reading every primary source I could find on how production agentic systems actually work — Anthropic's engineering blog, the Claude Code docs, the multi-agent research writeups, ArXiv's recent flood of agent-architecture papers, LangChain's context-engineering post, OpenTelemetry's GenAI conventions, Lilian Weng's failure taxonomy.

A pattern keeps showing up that should worry every PM building on top of LLMs.

The mental model most teams import from "system design" — request, response, deterministic logic, deploy and forget — quietly breaks when you point it at an agent. Not catastrophically. Subtly. Demos still work. Production still ships. But failures move into a different shape, and the PM who hasn't updated their model is the PM whose product gets surprised in front of users.

Here's what's actually different — and the three things AI PMs need to own that didn't exist on the old job description.

Part 01

Four shifts that change everything.

Every concept further down the page flows from these. They're not engineering nuances — each one moves a decision into the PM's column.

SHIFT 01 · BEHAVIOR

Deterministic → Probabilistic

same in = same out → it depends

Same input, same output, was the whole point of testing. With an LLM, temperature, retrieved context, conversation history, even prompt phrasing nudge the output.

PM action: design for acceptable unpredictability — confidence thresholds, fallback paths, hard-stop rules.
SHIFT 02 · DEPLOYMENT

Ship-and-done → Continuous feedback

freeze on deploy → drift over time

Models get updated. Retrieval quality degrades as your knowledge base ages. Prompt tweaks rot in ways nobody notices for weeks.

PM action: eval pipelines that run on every change, like CI runs on every commit. Block deploys when scores regress.
SHIFT 03 · DATA

Queried → Embedded into reasoning

data → user → data → cognition

Old world: your data pipeline affected what users saw. Agent world: it affects how the system thinks. A retrieved chunk shapes mid-task attention.

PM action: own context architecture as a product decision — what does the model need to know, when, and what expires?
SHIFT 04 · SAFETY

Input validation → Output guardrails

trust the logic → verify the output

SQL injection and type checks still matter. But agents can hallucinate, leak PII, or call tools they shouldn't — with completely valid inputs.

PM action: embed guardrails on the way out — content filters, tool gates, PII redactors. Governance by design.
Part 02 · The Pattern

The agentic loop: why "stateless" is the wrong mental model.

Traditional APIs are stateless. Request goes in, response comes out, done. Agents loop.

The pattern from the ReAct paper (Yao et al., 2022) — Thought → Action → Observation, repeat — is the architectural core of every production agent system worth building on. The model thinks, calls a tool, sees the result, thinks again, calls another tool, eventually decides it's done.

The base loop. Intelligence lives in the layers around it — context, tools, permissions — not in clever orchestration.

What's striking, when you read the recent Claude Code architecture paper (arXiv 2604.14228), is how simple this loop is in practice. It's a while-loop. The intelligence lives in the layers around it — the context that gets assembled each turn, the tools that are exposed, the permission rules that gate them — not in clever orchestration logic.

Good news: the architecture is comprehensible. It's not a black box.

Trap: a loop with no stop condition will burn your API budget in a single user session. Anthropic's Building Effective Agents guide is explicit that step counts are unpredictable and cost-latency trade-offs are a design decision — meaning a PM decision.

If you're not the one defining max-iteration limits, wall-clock timeboxes, and what happens when an agent hits them — then nobody is.

The failure mode shows up as a five-figure bill or a hung task.
Part 03 · The Skill Shift

Context is the product.

Prompt engineering is dead. Context engineering replaced it — and it landed in the PM's lap.

LangChain's July 2025 post on Context Engineering for Agents names four strategies any production agent has to handle:

  • Write — store context outside the window so it persists across turns.
  • Select — retrieve just-in-time. (RAG is one mechanism, not the only one.)
  • Compress — prune. Microsoft Research's LLMLingua shows up to 20× compression on real prompts.
  • Isolate — split context across specialized agents so each works with a focused, minimal window.

Why does this land in the PM's lap? Because you're the one who knows:

  • What the user expects the agent to remember between sessions — and what's a privacy violation if it does.
  • Which knowledge source is freshest, and which is dangerously stale.
  • When to expand context vs. when adding more would degrade the answer (see Liu et al.'s Lost in the Middle — long contexts produce U-shaped performance, with the middle suffering most).

A diagnostic when an agent misbehaves

Ask what layer of context broke. Each is a different bug with a different fix — wrong diagnosis, wrong fix, recurring failure.

Part 04 · The Job

The three things AI PMs have to own.

Bad model choice is recoverable. Bad prompt phrasing is recoverable. The three below are not. Get them wrong and the product fails in production at user cost.

Context Architecture

What does the model know about the user, the task, and the world? When does it know it? What expires? This is product, not infra. The choice to retrieve from a knowledge base vs. inject from a system prompt vs. summarize from past turns is a UX decision. Write it down.

Evaluation Infrastructure

Evals are your unit tests for AI products. Without them, every prompt change is an unmonitored production deploy. Start with the logs you've actually read. Grow it from real failures. Run on every change. Block deploys on regression.

Governance Model

What is the agent allowed to do? Not say — do. Read your database? Write to it? Send email? Spend money? If you can't answer "what's the worst this agent can do in one turn?" — you don't have a governance model yet.

A concrete pattern: Claude Code's permission model

Three tiers — evaluated in this order, with deny always taking precedence — plus PreToolUse hooks that intercept any call.

deny → match wins, tool blocked always wins
ask → match wins, prompt the user human in loop
allow → match wins, tool runs unprompted trusted ops

A practical PM exercise: list every tool your agent can invoke. Classify each one by blast radius — safe (read-only, idempotent), reversible (writes that can be undone), irreversible (sends a message, charges a card, deletes a record). Map approval gates to that classification. IBM's Agentic AI Governance Playbook frames the same idea at enterprise scale: governance moves from review-time to runtime.

You can hand any of these off to engineering. They will make the call without you. The call will not be wrong, exactly. It just won't reflect what your users need.

Engineers can't read users for you.
Part 05 · Production

Failures that don't show in demos.

Demos are easy. Production is where the patterns Lilian Weng catalogued back in 2023 keep showing up.

Infinite loops Agent calls the same tool repeatedly, never terminates. Common when the stop condition is implicit.
Context overflow Conversation history pushes past the window mid-task. Important early instructions get evicted.
Re-retrieval thrash Agent keeps fetching the same content with slightly different queries — wasting tokens and time.
Tool fan-out Too many parallel calls hit upstream rate limits. Fast in dev, brittle in prod.
Hallucinated tool calls Invoking tools that don't exist or passing invalid arguments. Forces a recovery path.
Lost-in-the-middle Information stuck mid-context gets ignored. Liu et al., 2023.

The fix isn't to write smarter prompts. It's to define tripwires as product requirements — explicit caps on iterations, tool calls, token budgets, wall-clock time. When a tripwire fires, the agent returns its best partial answer with calibrated uncertainty rather than thrashing harder.

The specific numbers depend on the use case. A coding agent might tolerate dozens of tool calls per task; a customer-facing chatbot might cap at three. The right number is the one your PM team has picked and written down, not the default the framework ships with.

Demand a trace, not a log

You cannot debug a probabilistic system from print statements. The emerging standard is OpenTelemetry's GenAI semantic conventions — span types like invoke_agent, execute_tool, create_agent, with attributes for token usage, tool definitions, conversation IDs.

Part 06 · Get Up To Speed

Your prioritized learning map.

If you came up through traditional product work and you're feeling behind, here's what to prioritize. P0 items are non-negotiable — the difference between collaborating with engineering and being a passenger.

P0
Context Engineering

RAG architectures, token budgeting, the four LangChain strategies, long-context degradation. Start with the LangChain post and Lost in the Middle.

P0
Agentic Architecture Patterns

Anthropic's canonical five workflow patterns from Building Effective Agents: prompt chaining, routing, parallelization, orchestrator-workers, evaluator-optimizer. The vocabulary your engineers are using.

P0
Evaluation Design

Hamel Husain's Your AI Product Needs Evals is the best single read. Then learn one platform — Braintrust, LangSmith, or Langfuse — well enough to read its dashboards.

P1
Governance & Permissions

Read Claude Code's permission docs and IBM's Agentic AI Governance Playbook. Map your own product's tools to a risk classification. Show the result to your engineering lead.

P1
Observability & Failure Literacy

Skim OpenTelemetry GenAI conventions until you can name the standard span types. Read Lilian Weng's failure taxonomy. Ask to see one production trace per week.

P2
Multi-Agent Coordination

Anthropic's multi-agent research writeup is the most concrete public reference. arXiv 2605.13850 extends the pattern set if you want depth.

P2
Traditional system design fundamentals

Still foundational. Distributed systems, caching, queues, contracts. AI systems sit on top of these, not instead of them.

Closing

What this all comes down to.

Most teams shipping AI products today are working with the right model, sensible prompts, and a roughly correct architecture. They still hit the same wall in production, because three things that used to be engineering concerns now live in the PM's column:

  • Context architecture — what the model knows, when, and for how long.
  • Evaluation infrastructure — how you know it works.
  • Governance — what the agent is allowed to do, and with what blast radius.

The PMs who internalize this — who treat context, evals, and governance as product artifacts they own — are the ones whose products don't surprise them in production.
